## Why

Per-change Playwright functional tests are generated by the planner but never executed pre-merge. The current architecture runs `test_command` (Jest) in the worktree and `smoke_command` (Playwright) post-merge on main. This causes three problems:

1. **Jest crashes on Playwright files** — Jest picks up `.spec.ts` files and tries to run them in jsdom, causing `TypeError: Class extends value undefined`. Agents waste tokens (1.57M in one run) without identifying the root cause.
2. **No pre-merge E2E validation** — Playwright tests only run after merge, so broken code lands on main with no change attribution.
3. **Consolidation anti-pattern** — The planner defers all Playwright tests to a single "e2e" change instead of each feature change owning its own `.spec.ts`. This consolidation change is too complex for one agent and creates a single point of failure.

## What Changes

### Orchestrator — new `e2e_command` directive (wt-tools)

- Add `e2e_command` to the verify gate flow in `verifier.sh`, running **pre-merge in the worktree** between build and scope check
- Port isolation: orchestrator sets `PW_PORT` env var per worktree to avoid port collisions between parallel changes
- Retry logic: same pattern as `test_command` (fail → resume agent with error context → retry)

### Planner — per-change E2E ownership (wt-tools)

- Update the "Functional test planning" section in `planner.sh` (both spec-mode and brief-mode) to instruct: each feature change MUST create its own `tests/e2e/<feature>.spec.ts` file — not defer to a consolidation change
- The first change that creates Playwright tests must also set up `playwright.config.ts` with dynamic port and add `testPathIgnorePatterns: ["/tests/e2e/"]` to `jest.config`
- Remove the "last change may consolidate all E2E tests" instruction

### Template rules — coexistence (wt-project-web)

- Add Jest/Playwright coexistence section to `testing-conventions.md`: `testPathIgnorePatterns`, dynamic port config, DB isolation patterns
- Update `smoke-test.yaml` directive: document that `smoke_command` is now optional (for cross-feature integration only), primary E2E coverage comes from `e2e_command`

### DB isolation — design for per-worktree databases (design only)

- SQLite: zero config — each worktree has its own `dev.db` file, `prisma db push` + seed works automatically
- PostgreSQL/MySQL: future work — `e2e_db_setup`/`e2e_db_teardown` hooks to create/drop per-worktree databases (e.g., `myapp_wt_cart_feature`)
- DB schema changes per-change: each worktree may have different Prisma schema (adding models), which is naturally isolated with file-based DBs
- Document the pattern in testing-conventions.md, implement PostgreSQL hooks later

## Capabilities

### New Capabilities

- `pre-merge-e2e`: Verify gate runs `e2e_command` (Playwright) in the worktree before merge, with port isolation and DB isolation. Each feature change owns its E2E tests.

_(none — no existing spec-level behavior changes)_
